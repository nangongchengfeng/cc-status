# ADR 0002: 首版服务端配置层使用标准库与环境变量

## 状态

已接受

## 日期

2026-05-21

## 背景

`server/CLAUDE.md` 对 Gin RESTful API 项目给出了通用偏好，其中配置层默认建议使用 `viper`。这个偏好在配置来源较多、存在配置文件分层、热更新或复杂绑定需求时是合理的。

但 `cc-status/server` 首版的实际配置需求非常小，目前只包含：

- 监听地址
- SQLite 路径
- 静态 Bearer token

同时，首版还存在几个明确约束：

- 鉴权 token 必填，未配置时服务端必须启动失败
- 配置来源越少越容易验证和排障
- 项目优先追求最小可维护实现，而不是提前引入复杂配置框架

如果在这个阶段引入 `viper`：

- 会增加依赖、初始化逻辑和调试面
- 会让“配置文件、环境变量、默认值”的优先级讨论先于真实需求落地
- 会偏离首版“标准库优先、最小实现”的整体取舍

## 决策

首版服务端配置层使用 Go 标准库加环境变量读取配置，不引入 `viper`。

当前约定如下：

- 使用 `os.Getenv` 读取运行时配置
- `CC_USAGE_SERVER_AUTH_TOKEN` 为必填项，缺失时启动失败
- `CC_USAGE_SERVER_LISTEN_ADDR` 和 `CC_USAGE_SERVER_SQLITE_PATH` 提供合理默认值，并允许环境变量覆盖

## 备选方案

### 立即引入 `viper`

优点：

- 更贴近 `server/CLAUDE.md` 中的通用偏好
- 如果后续很快出现多来源配置，迁移成本较低

缺点：

- 对当前极小配置集来说属于过度设计
- 增加依赖和隐性行为，不利于首版排障
- 会让简单启动路径变复杂

### 使用自定义配置文件加环境变量覆盖

优点：

- 能更早支持本地文件配置
- 便于集中查看配置项

缺点：

- 当前只有少量参数，收益不足
- 需要额外定义文件格式、路径和优先级
- 会把配置管理复杂度提前引入到首版

## 影响

正面影响：

- 启动路径清晰，配置行为容易解释
- 少量运行参数直接映射到环境变量，便于本地和 CI 使用
- `AuthToken` 必填这一安全约束可以在启动阶段直接落地

负面影响：

- 当前不支持配置文件
- 如果后续配置项明显增长，可能需要重构配置层
- 与 `server/CLAUDE.md` 的通用偏好存在一次经过记录的例外

## 落地说明

- 配置结构定义在 `server/internal/config/config.go`
- 当前使用标准库读取环境变量并填充默认值
- `CONTEXT.md` 明确记录了这是对 `server/CLAUDE.md` 的有意偏离，而不是遗漏实现

## 何时重审

当服务端出现以下任一情况时，重审是否迁移到更完整的配置方案：

- 配置项显著增多
- 需要配置文件、环境隔离或多层覆盖
- 需要统一管理比当前更多的第三方集成参数
