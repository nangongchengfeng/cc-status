# ADR 0001: 服务端核心域使用 `usage_reports`

## 状态

已接受

## 日期

2026-05-21

## 背景

服务端最早的参考材料和历史实现草稿沿用了 `proxy_request_logs` 这一命名。这个名字来自代理网关场景，天然会让维护者联想到以下语义：

- 表中的每一行都是代理层的原始请求日志
- 未来字段会围绕 HTTP 代理行为继续扩展，例如状态码、时延、流式片段或供应商网关字段
- 费用计算只是附着在代理日志上的副产物

但首版 `cc-status/server` 的真实职责并不是代理日志平台，而是接收客户端上报的 Claude Code token 使用数据，并把这些数据作为可计费、可统计、可追溯的**使用记录**保存下来。当前持久化字段也只覆盖上报、计费、查询和统计所需的最小集合，不包含代理语义下常见的网络层字段。

如果继续沿用 `proxy_request_logs`：

- 会误导后续实现者把服务端理解成“代理日志兼容层”
- 会弱化当前领域里的核心对象，也就是**使用记录**
- 会增加后续接口命名、文档表述和统计口径的不一致

## 决策

服务端核心事实表统一命名为 `usage_reports`，领域语言统一使用“使用记录”。

同步、统计、日志查询、定价计算和文档表述都围绕以下约定展开：

- 入库对象称为**使用记录**
- 核心表名为 `usage_reports`
- 不再在服务端领域中使用 `proxy_request_logs` 作为当前实现的正式命名

## 备选方案

### 继续沿用 `proxy_request_logs`

优点：

- 对照旧草稿和历史材料时迁移成本更低
- 表名看起来更接近某些既有实现

缺点：

- 与当前产品职责不符
- 容易诱导出无关的代理语义字段
- 让 `CONTEXT.md`、接口文档和代码命名更难保持一致

### 使用更泛化的名字，例如 `usage_events`

优点：

- 抽象程度更高，未来似乎更容易扩展

缺点：

- 事件语义过宽，无法直接表达“这是客户端上报并可计费的使用记录”
- 会削弱查询接口和统计接口中的业务表达

## 影响

正面影响：

- 代码、接口和文档围绕同一套领域语言组织，更容易维护
- 日志查询、统计排行和计费逻辑都能直接指向“使用记录”这一核心概念
- 后续如需引入代理层数据，也能作为新领域对象单独建模，而不是污染当前事实表

负面影响：

- 需要在阅读旧材料时做一次术语映射，把 `proxy_request_logs` 理解为历史草稿中的旧称
- 某些外部参考文档中的命名与当前代码不一致，需要显式说明

## 落地说明

- GORM 实体表名为 `usage_reports`
- `CONTEXT.md` 中服务端核心域语言使用“使用记录”“使用记录库”
- 查询和统计接口都基于 `usage_reports` 聚合

## 何时重审

只有在服务端目标明确扩展为“代理日志平台”，并且新的核心事实表需要承载网络代理语义字段时，才重审本决策。
