# Claude Usage Sync Server

## Problem Statement

当前仓库已经有一个 Go client，用于从本地 Claude Code **会话日志**中提取**Token 使用记录**并准备上报，但仓库中仍缺少一个可部署的中心服务来稳定接收这些**上报批次**、完成费用计算、持久化**使用记录**，并向查询端提供统计与排障 API。没有这个服务，团队无法把多台机器的 Claude Code 用量集中存储、统一计费、查询趋势，也无法对默认定价命中、重复上报、客户端排行等关键信息进行管理和排查。

## Solution

提供一个基于 `Gin + GORM(SQLite)` 的 Go 服务端，首版只覆盖 API，不做内置 Web 仪表盘。服务端接收来自 client 的 `POST /api/v1/sync` **上报批次**，对单条**使用记录**做幂等去重、模型定价匹配、费用计算和数据库写入，并提供总览统计、时间趋势、原始日志查询以及模型定价管理 API。服务端统一使用 Bearer token 保护除 `GET /healthz` 之外的所有 API，并在启动时自动建表、幂等初始化模型定价数据。

## User Stories

1. 作为平台管理员，我希望部署一个集中式服务端来接收多台电脑上报的 Claude 使用数据，以便统一统计团队成本。
2. 作为平台管理员，我希望服务端使用 Go 构建并运行在单个二进制进程中，以便部署和运维足够简单。
3. 作为开发者，我希望服务端技术栈使用 `Gin + GORM(SQLite)`，以便与仓库既定服务端约束保持一致。
4. 作为开发者，我希望服务端把已入库数据统一称为**使用记录**，而不是沿用代理场景命名，以便领域语言与真实职责一致。
5. 作为开发者，我希望核心事实表命名为 `usage_reports`，以便避免 `proxy_request_logs` 这种误导性的旧术语。
6. 作为运维人员，我希望首版服务端只提供 API，不捆绑仪表盘页面，以便先把接收、计费和查询闭环稳定跑通。
7. 作为调用方，我希望能够通过 `POST /api/v1/sync` 提交一个**上报批次**，以便一次发送多条**使用记录**。
8. 作为调用方，我希望 `POST /api/v1/sync` 的成功响应兼容现有 client 的字段格式，以便现有 client 无需修改即可联调。
9. 作为开发者，我希望 `POST /api/v1/sync` 继续返回 `code`、`message`、`accepted_count`、`duplicate_count`，以便兼容现有同步协议。
10. 作为调用方，我希望当请求头缺少或携带错误 Bearer token 时收到 `401`，以便明确知道是鉴权失败。
11. 作为平台管理员，我希望除 `GET /healthz` 外的所有 API 都统一受静态 token 保护，以便避免查询数据被匿名访问。
12. 作为平台管理员，我希望静态 token 是必填配置，服务端缺失该配置时直接启动失败，以便避免误把服务裸奔到网络上。
13. 作为开发者，我希望 `POST /api/v1/sync` 对批次级非法输入返回 `400`，以便尽早暴露调用方错误。
14. 作为开发者，我希望 `reports` 数组不能为空，以便空请求不会被误当成一次成功同步。
15. 作为开发者，我希望首版只接受 `app_type=claude`，以便先收敛到与现有 client 一致的单一来源。
16. 作为开发者，我希望首版只接受 `data_source=session_log`，以便保持来源语义整洁且可控。
17. 作为开发者，我希望 `client_id` 被视为透明的稳定技术标识，只校验非空和长度上限，以便兼容现有 client 的持久化标识。
18. 作为平台管理员，我希望使用 `(client_id, request_id)` 作为幂等去重键，以便不同客户端即使偶然产生相同 `request_id` 也不会相互冲突。
19. 作为平台管理员，我希望当 `(client_id, request_id)` 已存在时，该条记录被计入 `duplicate_count` 而不是报错，以便安全重试。
20. 作为平台管理员，我希望同一请求体内的重复 `request_id` 也按 duplicate 处理，以便 client 发生拼批错误时不会让整批失败。
21. 作为开发者，我希望单个 `POST /api/v1/sync` 请求在数据库层使用单事务，以便除了重复冲突以外的其他数据库错误都能整批回滚。
22. 作为开发者，我希望唯一键冲突之外的数据库异常会导致整个批次回滚并返回 `500`，以便客户端不会面对“接口失败但已落库一半”的模糊状态。

24. 作为平台管理员，我希望费用字段统一按 `USD` 存储，以便与已有 `*_cost_usd` 语义保持一致。

## Implementation Decisions

- 服务端首版采用 `Gin + GORM(SQLite)`，并遵循既有的 `handler/service/repository/model` 分层约束。
- 服务端核心领域使用**使用记录**、**上报批次**、**使用记录库**、**模型定价**、**默认定价**、**定价来源**等术语，不再沿用 `proxy_request_logs` 这类代理场景命名。
- 核心事实表命名为 `usage_reports`；首版只保留最小字段集：客户端标识、请求标识、来源字段、模型、token 计数字段、费用字段、`session_id`、`created_at`、`inserted_at`、`pricing_source` 等查询与计费必需字段。
- 首版只支持 `app_type=claude` 与 `data_source=session_log`；其他取值按参数错误处理。
- `POST /api/v1/sync` 是协议兼容特例：成功时返回 `{code, message, accepted_count, duplicate_count}`；除它之外的所有成功响应统一使用 `{ "data": ... }`。
- `POST /api/v1/sync` 对单个请求批次使用单事务。唯一键冲突 `(client_id, request_id)` 计入 `duplicate_count` 并继续；其他数据库错误回滚整批并返回 `500`。
- 同一请求体内若存在重复 `request_id`，服务端仍依赖统一的幂等规则处理，视为 duplicate 而不是 `400`。
- 顶层请求和单条 report 都执行严格校验：`client_id` 非空且长度受限，`reports` 至少一条，`created_at` 为 Unix 秒且满足宽松时间范围，来源字段必须命中白名单。
- `client_id` 在服务端是透明的稳定技术标识，不要求 UUID 格式，也不做机器名语义推断。
- 费用字段统一按 `USD` 计算和存储，不修改 `*_cost_usd` 命名。
- 模型定价匹配顺序固定为：精确匹配 `model_id == model`、最长前缀匹配、全局 `placeholder` 默认定价；首版不做 `contains` 匹配。
- `model_id` 与上报 `model` 统一先做小写规范化，再参与匹配、存储和返回。
- `pricing_source` 固定为三个枚举值：`exact`、`prefix`、`default`。
- `model_pricing` 必须允许存在一条全局 `placeholder` 默认定价，并且全局只能有一条；创建或更新冲突时返回 `409`。
- 首版模型定价管理 API 仅覆盖 `GET /api/v1/model-pricings`、`POST /api/v1/model-pricings`、`PUT /api/v1/model-pricings/:id`，不做删除接口。
- `PUT /api/v1/model-pricings/:id` 采用全量更新语义，不支持部分更新；未提供的可编辑字段不会隐式保留旧值。
- `GET /api/v1/model-pricings` 返回全部定价记录，并显式暴露 `is_placeholder`，默认定价不做隐藏。
- 查询 API 包括：
- `GET /api/v1/stats/overview`：返回总 token、总成本、总请求数、活跃客户端数、Top 模型和 Top 客户端。`top_models` 固定返回前 5 条，按 `tokens DESC, model ASC` 排序；`top_clients` 固定返回前 5 条，按 `total_cost_usd DESC, client_id ASC` 排序。
- `GET /api/v1/stats/trend`：仅支持 `interval=hour|day`，按 `Asia/Shanghai` 聚合，并补零返回完整时间轴。
- `GET /api/v1/logs`：采用 `offset` 分页，默认 `limit=20`、最大 `100`，默认排序 `created_at DESC, id DESC`，并返回完整 token 字段、完整费用字段和 `pricing_source`。
- 统计和列表 API 的成功响应遵循 `server/CLAUDE.md` 的通用包装；错误响应始终使用正确 HTTP 状态码。
- 服务端鉴权采用统一静态 Bearer token，除 `GET /healthz` 外所有 API 都需要鉴权。静态 token 为必填配置，未配置时服务端拒绝启动。
- 首版服务端配置层显式偏离 `server/CLAUDE.md` 的 `viper` 偏好，改用 Go 标准库加环境变量读取配置。这是一个有意记录的架构取舍。
- 首版环境变量至少包括：监听地址、SQLite 路径、静态 token。默认 SQLite 路径为 `./server/data/server.db`，允许通过环境变量覆盖。
- 首版采用启动时自动建表与幂等初始化数据，不单独引入外部 migration 工具；常见 Claude 模型定价和一条全局默认定价在启动流程中初始化。
- 建议的深模块边界包括：
- 同步接收模块：封装批次校验、幂等写入、事务和兼容响应。
- 定价模块：封装模型名规范化、匹配优先级、费用计算和 `pricing_source` 决策。
- 统计查询模块：封装总览、趋势、分页日志查询及补零逻辑。
- 模型定价管理模块：封装 placeholder 唯一性校验和全量更新规则。
- 配置模块：封装环境变量解析、默认值和必填配置校验。
- 由于“服务端核心域改名”和“首版配置层不用 `viper`”都属于偏离既有预期的硬决策，后续实现时应考虑补充 ADR 记录原因。

## Testing Decisions

- 好的测试只验证外部行为和业务契约，不绑定内部实现细节。例如验证“重复记录会计入 `duplicate_count` 并保持幂等”“最长前缀定价是否正确命中”“趋势接口是否补零”，而不是断言 repository 内部调用顺序。
- 首版优先测试四类模块：同步接收模块、定价模块、统计查询模块、模型定价管理模块。
- 同步接收测试应覆盖：鉴权失败、空批次、非法来源、批内重复、库内重复、事务回滚、成功响应兼容字段、`accepted_count + duplicate_count` 覆盖整批等关键路径。
- 定价模块测试应覆盖：模型名小写规范化、精确匹配、最长前缀匹配、默认定价回退、`pricing_source` 枚举、placeholder 唯一性冲突。
- 统计查询测试应覆盖：总览排行排序规则、趋势按 `Asia/Shanghai` 聚合、空桶补零、分页日志默认排序、时间过滤和完整费用字段返回。
- 模型定价管理测试应覆盖：`GET` 返回全部记录含 `is_placeholder`、`POST` 创建普通定价、创建第二条 placeholder 返回 `409`、`PUT` 全量更新和更新为 placeholder 的冲突场景。
- repository 层适合使用 SQLite 集成测试，因为项目本身就以 SQLite 作为首版数据库；这能以最接近真实行为的方式验证唯一键、事务和聚合 SQL。
- handler 层适合使用 `httptest` 验证鉴权、参数绑定、HTTP 状态码与响应结构。
- service 层适合聚焦业务契约测试，特别是同步事务语义、费用计算、趋势补零与默认定价决策。
- 当前仓库的 client 侧已经采用 Go 标准测试和 SQLite 实库测试模式，server 侧可以沿用同类风格，避免引入不必要的测试框架复杂度。

## Out of Scope

- 首版不实现内置 Web 仪表盘、模板页面、Chart.js 或 ECharts 集成。
- 首版不接入 Codex、Gemini 或其他 `app_type` / `data_source`。
- 首版不实现用户体系、RBAC、多租户或动态 token 管理。
- 首版不实现模型定价删除、批量导入、审计日志或回滚工具。
- 首版不引入外部 migration 工具、专门的 migrate CLI 或版本化 schema 管理流程。
- 首版不引入 MySQL/PostgreSQL 兼容层，数据库先固定为 SQLite。
- 首版不实现客户端投影表 `clients`；所有客户端统计和列表均从 `usage_reports` 聚合得到。
- 首版不实现 cursor 分页、任意趋势粒度、开放式来源白名单或模糊 `contains` 定价匹配。
