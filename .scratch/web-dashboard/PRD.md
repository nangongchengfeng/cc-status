# Claude Usage Web Dashboard

## Problem Statement

当前仓库已经有一个可运行的 Go 服务端和 SQLite 数据模型，能够接收 Claude Code 的**使用记录**并提供基础查询 API，但前端 `web/` 还没有真正落地成一个可展示的数据大屏。同时，现有通用统计接口不足以完整支撑前端大屏对 token 细分趋势、缓存效益分析和 Top 10 排行的展示需求。作为求职 demo，如果没有一个打开即见的现代化 Web 仪表盘，项目很难完整体现“后端数据服务 + 前端可视化产品”的全栈能力。

## Solution

在仓库根目录实现一个基于 `React + TypeScript + Vite + Tailwind CSS + Recharts + TanStack Query` 的单页数据大屏，首页路由直接使用 `/`。前端直接对接服务端的 dashboard 聚合接口和现有日志分页接口，展示总览指标、统一时间轴趋势图、模型排行、客户端排行、缓存效益分析和最近请求列表。页面采用深色科技风和自由网格布局，优先适配桌面演示场景，并由前端负责时间预设、金额与时间格式化以及图表交互细节。

## User Stories

1. 作为项目展示者，我希望有一个完整的 Web 大屏，以便在求职时展示全栈开发能力。
2. 作为用户，我希望打开网站首页 `/` 就直接进入仪表盘，以便演示时无需额外跳转。
3. 作为用户，我希望打开页面就能看到 Claude Code 使用总览，以便快速了解整体消耗。
4. 作为用户，我希望看到总 Token 消耗、总费用、总请求数、活跃客户端数四个核心指标卡片，以便一眼掌握关键数据。
5. 作为用户，我希望看到费用趋势图，以便了解成本随时间的变化情况。
6. 作为用户，我希望看到 Token 使用趋势图，以便区分输入、输出、缓存读、缓存创建的比例。
7. 作为用户，我希望趋势图基于统一时间轴联动展示，以便不同图表之间容易横向对比。
8. 作为用户，我希望看到模型使用排行（TOP 10），以便知道哪些模型最常用。
9. 作为用户，我希望模型排行优先显示更友好的展示名，以便大屏阅读体验更好。
10. 作为用户，我希望展示名缺失时仍然能看到原始模型标识，以便不丢失排障信息。
11. 作为用户，我希望看到客户端费用排行（TOP 10），以便知道哪台客户端标识花费最高。
12. 作为用户，我希望看到缓存效益分析，以便了解使用缓存节省了多少费用。
13. 作为用户，我希望缓存效益同时展示节省金额和缓存建设成本，以便完整理解缓存收益与代价。
14. 作为用户，我希望看到最近请求列表，以便查看最新的**使用记录**。
15. 作为用户，我希望最近请求列表直接反映服务端排序后的结果，以便看到最新到最旧的记录顺序。
16. 作为用户，我希望可以选择时间范围（今天、最近 7 天、最近 30 天、本月、全部），以便查看不同时间段的数据。
17. 作为开发者，我希望这些时间范围只存在于前端预设中，以便后端接口保持通用。
18. 作为用户，我希望切换时间范围后所有卡片和图表一起刷新，以便保持页面口径一致。
19. 作为用户，我希望图表支持鼠标悬停查看具体数值，以便获取更详细的信息。
20. 作为用户，我希望页面采用深色科技风主题，以便看起来像专业的监控大屏。
21. 作为用户，我希望页面布局采用清晰的卡片和网格结构，以便核心信息有明显层级。
22. 作为用户，我希望大屏优先适配 1920x1080 分辨率，以便在会议室或外接显示器上展示效果最佳。
23. 作为用户，我希望页面在较窄桌面窗口下也能基本可用，以便日常开发调试不受阻。
24. 作为开发者，我希望前端技术栈使用 `React + TypeScript + Tailwind CSS + Recharts`，以便兼顾工程可维护性和视觉定制能力。
25. 作为开发者，我希望使用 TanStack Query 管理服务端状态，以便优雅处理请求、缓存和 loading/error 状态。
26. 作为开发者，我希望 API 调用封装成自定义 hooks，以便页面组件只关心数据使用而非请求细节。
27. 作为开发者，我希望 dashboard 数据和最近请求分别请求不同资源，以便遵循服务端职责边界。
28. 作为开发者，我希望前端只消费后端原始业务字段，并自行做金额、时间和标签格式化，以便页面表现更灵活。
29. 作为开发者，我希望项目结构清晰地区分 pages、components、hooks、api、types 和 utils，以便展示代码组织能力。
30. 作为用户，我希望页面有明确的 loading 状态，以便知道数据正在加载中。
31. 作为用户，我希望请求失败时有错误提示和可恢复的页面状态，以便知道哪里出了问题。
32. 作为维护者，我希望大屏首版只做单页，不引入复杂路由和后台管理逻辑，以便聚焦展示价值。

## Implementation Decisions

- 前端项目使用单页应用方案，首页路由直接为 `/`，不再额外引入 `/dashboard` 路由层级。
- 技术栈采用 `React 18 + TypeScript 5 + Vite + Tailwind CSS + Recharts + TanStack Query`，与当前 `web/CLAUDE.md` 保持一致。
- 页面采用功能模块化设计：页面路由放入 `pages/`，跨页面复用组件放入 `components/`，请求函数放入 `api/`，查询 hooks 放入 `hooks/`，类型定义放入 `types/`，格式化和图表辅助逻辑放入 `utils/`。
- 大屏主页面由以下模块构成：时间范围筛选器、四个总览卡片、趋势图区域、模型排行、客户端排行、缓存效益分析、最近请求表格。
- dashboard 数据和最近请求列表分别请求不同资源：前者请求服务端 dashboard 聚合接口，后者复用现有 `GET /api/v1/logs`。
- 时间范围选择器支持：今天、最近 7 天、最近 30 天、本月、全部；这些仅作为前端常量预设存在，再转换成后端需要的 `start_at`、`end_at`、`interval`。
- 默认时间范围使用最近 7 天；当时间跨度较短时优先请求小时粒度，跨度较长时优先请求天粒度。
- 趋势相关图表共享同一份统一时间桶数组数据源，避免前端维护多条独立时间轴。
- 图表组件统一使用 Recharts：
  - 费用趋势使用 `LineChart`
  - Token 细分趋势使用 `AreaChart`
  - 模型排行使用横向 `BarChart`
  - 客户端排行使用横向 `BarChart`
  - 缓存效益可使用 `PieChart`、进度环或卡片式数值组合，但必须突出节省金额和建设成本
- 模型相关展示优先使用后端返回的 `display_name`，缺失时回退到原始 `model`。
- 客户端排行和最近请求中的客户端字段直接展示 `client_id`，首版不引入别名管理。
- 前端负责所有展示格式化，包括：金额缩写与小数位控制、UTC/上海时间标签格式化、长标签截断、Tooltip 文案和卡片辅助文案。
- 认证 token 写死在前端配置中，默认使用 `dev-token`，仅用于本地演示，不做登录流程。
- API 基址通过环境变量配置，默认指向本地服务端。
- 页面布局优先面向桌面大屏：总览卡片位于页面顶部，趋势图占据中部主要视觉区域，排行和最近请求位于下方或侧边辅助区。
- 视觉风格采用深色科技风：深色背景、卡片化层次、蓝紫橙绿系数据色，突出图表对比度和演示冲击力。
- 首版不做 mock 数据模式，直接依赖真实服务端数据；若服务端无数据，页面展示空态而不是伪造样本。

## Testing Decisions

- 好的测试只验证用户能观察到的页面行为和数据契约，例如时间范围切换后请求参数变化、loading/error/success 状态切换，以及表格和图表数据源映射正确，不测试 Tailwind 或 Recharts 本身。
- 首版优先测试自定义 hooks、时间范围转换工具和少量关键页面状态，而不是为每个视觉组件都写细碎快照测试。
- hooks 测试应覆盖：dashboard 数据请求、最近请求请求、query key 随时间范围变化、错误状态透传。
- 工具函数测试应覆盖：时间范围预设到 `start_at/end_at/interval` 的转换、金额格式化、模型展示名回退逻辑。
- 页面测试应覆盖：初始加载态、接口失败态、空数据态、成功渲染后的关键卡片和表格标题存在。
- 若测试成本过高，首版可接受以 `build` 验证和手工 smoke 为主，但至少应保证关键数据转换逻辑可被自动化测试。

## Out of Scope

- 移动端完美适配（只做基础响应式）。
- 用户登录/权限系统（使用写死的 token）。
- 数据导出功能。
- 告警/通知功能。
- 模型定价管理 UI。
- 同比/环比对比视图。
- 点击排行项进行筛选的交互。
- 多页面后台管理、权限页和设置页。
- mock 数据模式和设计稿编辑器。
- 完整 E2E 测试体系。

## Further Notes

- 该 PRD 依赖服务端新增 dashboard 聚合接口，因此不再以“只复用现有 overview/trend API”为前提。
- `web` 的技术约定已经更新为 `Tailwind CSS + Recharts`，不再沿用旧的 `Ant Design` 草稿方向。
- 作为求职 demo，页面应优先追求第一眼的视觉完成度和演示体验，但仍需保持代码结构清晰、命名稳定和前后端契约明确。
- 后续如需进入 issue 拆分，建议按“前端脚手架”“API 客户端与 hooks”“仪表盘主页面布局”“图表与表格组件”“空态与错误态打磨”几个垂直切片推进。
