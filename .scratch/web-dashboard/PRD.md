# Claude Usage Web Dashboard

## Problem Statement

当前仓库已经有一个完整的 Go 服务端，提供了总览统计、趋势查询、日志列表和模型定价管理的 API，但缺少一个可视化 Web 仪表盘来展示这些数据。作为一个求职 demo，仅有后端 API 不够完整，需要一个现代化的数据大屏来展示全栈开发能力，让数据更直观、更有冲击力。

## Solution

在仓库根目录新增 `web/` 文件夹，使用 React + TypeScript + Tailwind CSS + Recharts 构建一个深色科技风的数据大屏。大屏直接连接本地服务端 `http://127.0.0.1:8080` 的 API，展示总览指标、费用趋势、Token 使用趋势、模型排行、客户端排行、缓存效益分析和最近请求列表，并支持按时间范围筛选。认证 token 写死在代码里方便演示。

## User Stories

1. 作为项目展示者，我希望有一个完整的 Web 大屏，以便在求职时展示全栈开发能力。
2. 作为用户，我希望打开页面就能看到 Claude Code 使用总览，以便快速了解整体消耗。
3. 作为用户，我希望看到总 Token 消耗、总费用、总请求数、活跃客户端数四个核心指标卡片，以便一眼掌握关键数据。
4. 作为用户，我希望看到费用趋势图，以便了解成本随时间的变化情况。
5. 作为用户，我希望看到 Token 使用趋势图，以便区分输入、输出、缓存读、缓存创建的比例。
6. 作为用户，我希望看到模型使用排行（TOP 10），以便知道哪些模型最常用。
7. 作为用户，我希望看到客户端费用排行（TOP 10），以便知道哪台机器花费最高。
8. 作为用户，我希望看到缓存效益分析，以便了解使用缓存节省了多少费用（这是亮点）。
9. 作为用户，我希望看到最近请求列表，以便查看最新的使用记录。
10. 作为用户，我希望可以选择时间范围（今天、最近 7 天、最近 30 天、本月、全部），以便查看不同时间段的数据。
11. 作为用户，我希望选择时间范围后所有图表联动更新，以便保持数据一致性。
12. 作为用户，我希望图表支持鼠标悬停查看具体数值，以便获取更详细的信息。
13. 作为用户，我希望页面采用深色科技风主题，以便看起来像专业的监控大屏。
14. 作为开发者，我希望使用 React + TypeScript，以便展示现代前端技术栈的掌握能力。
15. 作为开发者，我希望使用 Tailwind CSS，以便快速实现美观的 UI。
16. 作为开发者，我希望使用 Recharts，以便实现丰富的图表效果。
17. 作为开发者，我希望使用 React Query，以便优雅地处理 API 请求、缓存和 loading/error 状态。
18. 作为开发者，我希望 API 调用封装成自定义 hooks，以便代码更清晰、复用性更好。
19. 作为用户，我希望页面有基础响应式设计，以便在不同大小的屏幕上都能正常显示。
20. 作为用户，我希望大屏优先适配 1920x1080 分辨率，以便在会议室或外接显示器上展示效果最佳。
21. 作为开发者，我希望 TypeScript 类型定义与后端 DTO 对应，以便保证前后端数据结构一致。
22. 作为用户，我希望认证 token 写死在代码里，以便演示时不需要额外配置。
23. 作为用户，我希望页面布局采用经典网格布局，以便模块清晰、信息层次分明。
24. 作为用户，我希望总览卡片在页面最上方，以便第一时间看到核心指标。
25. 作为用户，我希望趋势图占据较大空间，以便更好地观察数据变化。
26. 作为用户，我希望排行使用条形图或柱状图展示，以便直观对比。
27. 作为用户，我希望缓存效益使用饼图展示，以便直观看到缓存节省的比例。
28. 作为用户，我希望最近请求使用表格展示，以便查看详细信息。
29. 作为开发者，我希望项目结构清晰（components、hooks、types），以便展示代码组织能力。
30. 作为开发者，我希望有统一的 API 客户端封装，以便集中处理请求头、错误处理等逻辑。
31. 作为用户，我希望页面加载时有 loading 状态，以便知道数据正在加载中。
32. 作为用户，我希望请求失败时有错误提示，以便知道哪里出了问题。

## Implementation Decisions

- 前端项目放在仓库根目录 `web/`，与 `server/`、`client/` 平级。
- 技术栈：React 18 + TypeScript 5 + Vite + Tailwind CSS 3 + Recharts 2 + TanStack Query (React Query) 5。
- 采用功能模块化设计，分为：
  - `types/`：TypeScript 类型定义，与后端 DTO 保持一致。
  - `hooks/`：自定义 hooks，封装 API 调用（useOverview、useTrend、useLogs、useModelPricings）。
  - `components/`：React 组件，按功能拆分（MetricCard、CostTrendChart、TokenTrendChart、ModelRankChart、ClientRankChart、CacheBenefitChart、RecentLogsTable、TimeRangeSelector）。
  - `api/`：API 客户端封装，统一处理请求头、错误处理。
- 布局采用 CSS Grid + Tailwind 响应式类：
  - 默认大屏（1920x1080）：4 列总览卡片 + 2x2 图表网格 + 右侧最近请求表格。
  - 中等屏幕（1366x768）：2 列总览卡片 + 堆叠图表布局。
- 配色方案（深色科技风）：
  - 背景：`#0f172a`（slate-900）
  - 卡片背景：`#1e293b`（slate-800）
  - 主色调：`#3b82f6`（blue-500）
  - 输入 Token：`#a855f7`（purple-500）
  - 输出 Token：`#f97316`（orange-500）
  - 缓存读：`#22c55e`（green-500）
  - 缓存创建：`#06b6d4`（cyan-500）
  - 文字：`#e2e8f0`（slate-200）
- 时间范围选择器支持：今天、最近 7 天（默认）、最近 30 天、本月、全部。
- 时间范围变更时，所有图表数据联动刷新。
- 认证方式：Bearer token 写死在代码中，默认使用 `dev-token`。
- API 基址：默认 `http://127.0.0.1:8080`。
- 核心 API 对接：
  - `GET /api/v1/stats/overview` → 总览数据
  - `GET /api/v1/stats/trend` → 趋势数据
  - `GET /api/v1/logs` → 日志列表
  - `GET /api/v1/model-pricings` → 模型定价（可选展示）
- 图表组件使用 Recharts：
  - 费用趋势：`LineChart`
  - Token 趋势：`AreaChart`（堆叠区域图）
  - 模型排行：`BarChart`（横向条形图）
  - 客户端排行：`BarChart`（横向条形图）
  - 缓存效益：`PieChart` + 节省金额展示
- 最近请求使用表格展示，包含：时间、模型、输入/输出 Token、总费用、客户端。
- 没有 mock 数据功能，直接使用服务端真实数据。

## Testing Decisions

- 由于这是一个求职 demo 项目，优先保证功能完整性和视觉效果，不强制要求单元测试。
- 如后续需要测试，可考虑：
  - 测试自定义 hooks 的返回值和状态变化（使用 React Query 的测试工具）。
  - 测试组件在 loading/error/success 三种状态下的渲染。
  - 使用 Storybook 展示各个组件的不同状态。

## Out of Scope

- 移动端完美适配（只做基础响应式）。
- 用户登录/权限系统（使用写死的 token）。
- 数据导出功能。
- 告警/通知功能。
- 模型定价管理 UI（只展示，不提供编辑）。
- 同比/环比对比视图。
- 点击排行项进行筛选的交互。
- 单元测试和 E2E 测试（demo 优先）。

## Further Notes

- 项目使用 Vite 作为构建工具，比 Create React App 更快、更现代。
- 保持与现有 server 的 API 契约一致，不修改后端。
- 作为求职 demo，代码要注重可读性、结构清晰、注释适当（但不过度）。
- 可以在 README 中添加启动说明和截图，方便面试官快速了解。
