# Claude Usage Sync Client

## Problem Statement

当前需要一个独立于现有 Rust/Tauri 应用之外的 Go client，部署在每台安装了 Claude Code 的电脑上，负责从本地**会话日志**中增量提取**Token 使用记录**，并把新记录可靠地同步到中心服务。现阶段缺少一个可单独运行、可重试、可追踪、可跨平台部署的同步工具，导致本地 Claude Code 使用量无法统一采集。

## Solution

提供一个单次执行的 Go CLI 工具，首版聚焦 Claude Code 的 `~/.claude/projects/` JSONL **会话日志**。工具扫描主会话与 `subagents` 日志，按既有 Rust 逻辑完成过滤、单文件内去重、增量读取、批量上报和本地状态维护。工具使用 SQLite 统一持久化**同步状态**、**已上报记录**与**客户端标识**，默认把配置和数据库放到 `~/.cc-usage-client`。同时提供 `dry-run` 用于本地验证解析结果而不推进业务状态。

## User Stories

1. 作为运维人员，我希望在每台安装了 Claude Code 的电脑上运行一个单次 CLI，同步本机 Claude 使用数据，以便集中统计团队用量。
2. 作为开发者，我希望工具自动扫描 `~/.claude/projects/` 及其 `subagents` 子目录中的 JSONL 文件，以便不需要手工指定每个会话文件。
3. 作为开发者，我希望工具严格只提取满足既有规则的 assistant 消息，以便与现有 Rust 统计逻辑保持一致。
4. 作为开发者，我希望工具只接受存在 `message.stop_reason` 且 `output_tokens > 0` 的消息，以便过滤未完成或无效的流式中间状态。
5. 作为开发者，我希望同一文件内同一 `message.id` 出现多次时，优先保留带 `stop_reason` 的版本，以便保留最终完成的消息。
6. 作为开发者，我希望在同一文件内如果多个重复消息都满足 `stop_reason`，选择 `output_tokens` 更大的版本，以便匹配 Rust 侧的去重策略。
7. 作为开发者，我希望每条**Token 使用记录**都带有 `request_id = session:<message_id>`，以便客户端和服务端都能稳定去重。
8. 作为平台管理员，我希望 client 首次运行时自动生成稳定的**客户端标识**，以便同一台电脑后续上报能被服务端持续识别。
9. 作为平台管理员，我希望**客户端标识**被持久化，以便重启或升级工具后不会变化。
10. 作为开发者，我希望工具为每个**会话日志**维护独立的**同步状态**，以便只读取新增行而不是每次全量扫描。
11. 作为开发者，我希望**同步状态**至少记录 `last_modified` 和 `last_line_offset`，以便准确判断哪些行是新增内容。
12. 作为开发者，我希望文件未变化时直接跳过，以便降低常规同步时的 I/O 成本。
13. 作为开发者，我希望检测到文件截断、替换或行号回退时自动从头重扫该文件，以便避免漏掉新文件中的有效记录。
14. 作为开发者，我希望末尾半写入或坏 JSON 行被视为瞬时错误，以便不会错误消费未写完的日志内容。
15. 作为开发者，我希望当前文件出现瞬时错误时不推进该文件的**同步状态**，以便下次重试能重新读取完整内容。
16. 作为平台管理员，我希望上报失败时工具采用严格重扫策略，以便只有服务端确认成功的数据才会推进状态。
17. 作为平台管理员，我希望工具对成功上报的记录写入**已上报记录**，以便后续重试时不会重复发送相同 `request_id`。
18. 作为平台管理员，我希望本地状态统一保存在 SQLite 中，以便查询、并发控制和后续扩展更简单。
19. 作为平台管理员，我希望工具默认把配置、SQLite 和锁文件都放在 `~/.cc-usage-client`，以便部署路径稳定且易于排查。
20. 作为开发者，我希望 `config.yaml` 是主配置来源，但可以用环境变量覆盖关键配置，以便兼顾运维便利和敏感信息安全。
21. 作为运维人员，我希望服务端 URL、认证 token、批大小和超时时间可配置，以便接入不同环境。
22. 作为运维人员，我希望认证 token 可以通过环境变量覆盖配置文件，以便避免把敏感值写入磁盘。
23. 作为平台管理员，我希望工具以固定批大小分批发送记录，以便控制请求体大小并降低单次失败影响范围。
24. 作为平台管理员，我希望默认批大小为 500，以便在稳定性和效率之间取得平衡。
25. 作为平台管理员，我希望每批请求都有超时控制，以便网络异常时能及时失败并重试。
26. 作为平台管理员，我希望 HTTP 5xx 或网络错误触发指数退避重试，最多 3 次，以便提高短暂故障下的成功率。
27. 作为平台管理员，我希望服务端返回 HTTP 200 但业务码异常时整批按失败处理，以便在没有逐条回执的前提下避免误标已上报。
28. 作为平台管理员，我希望 `accepted_count + duplicate_count` 不能覆盖整批记录时整批重试，以便保证客户端状态只基于明确确认推进。
29. 作为运维人员，我希望同步命令在出现部分失败时返回非 0 退出码，以便计划任务系统能准确告警。
30. 作为运维人员，我希望成功但本次无新增记录时仍返回 0，以便区分“无数据”与“失败”。
31. 作为开发者，我希望同一台机器同一时刻只允许一个同步实例运行，以便避免竞争推进**同步状态**或重复上报。
32. 作为开发者，我希望使用进程级互斥锁阻止并发运行，以便让单次 CLI 语义保持简单可预测。
33. 作为开发者，我希望 `dry-run` 能复用真实解析和批次构造逻辑，以便本地验证尽量接近正式同步。
34. 作为开发者，我希望 `dry-run` 不推进**同步状态**和**已上报记录**，以便调试不会消耗真实待同步数据。
35. 作为开发者，我希望 `dry-run` 在数据库或**客户端标识**尚未初始化时允许创建基础依赖，以便首次部署就能做本地验证。
36. 作为开发者，我希望工具输出清晰的扫描文件数、提取记录数、上报成功数、跳过数和错误信息，以便快速定位同步问题。
37. 作为未来服务端开发者，我希望首版 client 预留明确的上报请求结构，以便后续 server 可以按约定实现接收接口。
38. 作为未来项目维护者，我希望 client 代码按照配置、状态存储、日志解析、同步编排和 HTTP 上报分层，以便各模块可独立测试和替换。
39. 作为未来项目维护者，我希望先在仓库里拆出 `client` 和 `server` 顶层目录，以便后续客户端和服务端可以并行演进。
40. 作为测试工程师，我希望核心行为可以通过自动化测试验证，而不是依赖手工检查日志，以便降低回归风险。

## Implementation Decisions

- 仓库结构先拆出顶层 `client` 与 `server` 两个目录，其中本次 PRD 只覆盖 client 首版实现，server 目录先作为后续中心服务的预留位置。
- client 第一版采用单次执行 CLI，而不是常驻进程；同步命令完成一次扫描、解析、上报后退出。
- CLI 至少提供两个入口能力：正式同步的 `sync`，以及仅验证解析与批次构造但不推进业务状态的 `dry-run`。
- 首版仅覆盖 Claude Code 的**会话日志**，即 `~/.claude/projects/` 下的主会话 JSONL 和 `subagents/*.jsonl`；Codex 与 Gemini 不在本轮实现范围。
- 日志解析规则严格对齐现有 Rust 逻辑：仅处理 `type == "assistant"`、存在 `message.stop_reason`、且 `output_tokens > 0` 的消息。
- 同一 `message.id` 的合并范围限定为单个文件内；优先保留有 `stop_reason` 的记录，若候选记录都满足该条件，则保留 `output_tokens` 更大的记录。
- 每条**Token 使用记录**的唯一标识为 `request_id = "session:" + message_id`，作为客户端去重和服务端幂等的基础。
- 本地持久化统一采用 SQLite；至少包含 `sync_state`、`reported_ids` 和用于保存**客户端标识**的元数据存储。
- `sync_state` 以文件绝对路径为键，保存文件纳秒级修改时间和最后处理到的行号，用于增量扫描。
- 只有当某个文件涉及的记录全部成功完成上报确认后，才推进该文件对应的**同步状态**；文件级推进与“严格重扫”语义保持一致。
- 整次运行的提交语义为按成功部分提交：已经明确成功的批次写入**已上报记录**，已经完整成功的文件推进**同步状态**；失败的批次和文件在下次运行中重试。
- 如果检测到文件被截断、替换或当前总行数小于已保存的行号，则将该文件视为已重置，并从头开始重扫。
- 如果读取文件时在尾部遇到半写入或 JSON 解析失败，则认定为瞬时错误；停止处理该文件，记录错误，并且不推进该文件的**同步状态**。
- 配置采用 `config.yaml` 为主、环境变量覆盖的策略；服务端 URL、认证 token、请求超时、批大小等都应支持配置。
- **应用数据目录**统一为 `~/.cc-usage-client`，配置文件、SQLite 数据库、锁文件以及其他运行元数据都放在该目录下。
- **客户端标识**采用首次本地生成 UUID 并持久化的方式，避免依赖机器指纹或硬件信息。
- 上报接口按照既定请求格式预留，单批固定为最多 500 条记录；服务端响应只有在 HTTP 成功且业务确认覆盖整批时才视为该批成功。
- 对网络错误、超时和 HTTP 5xx 使用指数退避重试，最多 3 次；如果业务响应码异常或确认数量不足，也按整批失败处理。
- 进程级互斥锁属于首版必备能力，用于保证同一台机器同一时刻只有一个 client 实例执行同步。
- 推荐的 client 内部模块边界为：配置与运行时模块、SQLite 状态存储模块、Claude 日志扫描解析模块、同步编排模块、HTTP 上报客户端模块、CLI 命令层。
- 模块设计优先抽成“深模块”：例如日志扫描解析模块应对外暴露稳定的扫描/提取接口，把文件发现、增量读取、单文件内去重和坏行处理封装起来；同步编排模块应统一管理批次发送、部分提交和错误归集。

## Testing Decisions

- 好的测试应聚焦外部行为和业务契约，而不是内部实现细节；例如验证“给定某类 JSONL 输入能产出哪些**Token 使用记录**、推进哪些**同步状态**、在失败时保留哪些状态”，而不是断言具体内部循环或私有函数调用顺序。
- 首版优先覆盖三个测试重点模块：Claude 日志扫描解析、SQLite 状态存储、同步编排。
- Claude 日志扫描解析测试应覆盖：主会话与 `subagents` 文件发现、assistant 过滤、`stop_reason` 过滤、`output_tokens > 0` 过滤、单文件内重复 `message.id` 合并、尾部坏行处理、文件截断后从头重扫。
- SQLite 状态存储测试应覆盖：数据库初始化、`sync_state` 读写、`reported_ids` 写入与查询、`client_id` 持久化、并发实例下的锁冲突行为。
- 同步编排测试应覆盖：按 500 条分批、成功部分提交、失败批次不推进状态、HTTP 5xx 与超时重试、`HTTP 200 + code != 0` 按失败处理、确认数量不足时整批失败。
- `dry-run` 测试应至少验证：允许初始化基础依赖，但不会写入**同步状态**或**已上报记录**。
- 配置层自动化测试不是首版必选重点，但应保留可测试接口，便于后续补充 `config.yaml` 与环境变量覆盖行为。
- 由于仓库当前没有现成的 Go 测试先例，测试策略应采用 Go 标准测试框架与临时目录/临时数据库驱动，尽量保持隔离、无外部依赖、可跨平台执行。

## Out of Scope

- 中心服务端的真实实现、数据库 schema 和部署方案。
- Codex、Gemini 或其他来源的会话日志采集。
- 费用计算、模型定价、代理日志与会话日志的跨数据源去重。
- 常驻后台进程、文件系统 watch、系统服务注册或定时任务安装器。
- 远程配置下发、多租户鉴权、数据加密传输之外的更复杂安全策略。
- 图形界面、Web 管理页或桌面集成。

## Further Notes

- 当前仓库还没有 Go 工程骨架，因此首版实现需要同时建立基础项目结构。
- 由于服务端尚未实现，client 设计应尽量把 HTTP 上报接口与同步编排解耦，方便后续对接 mock 服务和正式 server。
- 现有 [claude_usage_stats.md](file:///d:/Code/AI/cc-status/claude_usage_stats.md) 提供了 Rust 侧 Claude 解析规则的主要依据；首版实现需要以其中的 Claude 部分为行为基线，而不是复刻其中暂不需要的 Codex/Gemini 逻辑。
